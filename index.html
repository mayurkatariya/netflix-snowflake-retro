Case 01376791 — External Partner (VideoAmp) Access Blocked

Opened: June 12, 2026 Contact: Ankit Jain (ankitj@netflix.com) Jira: SNOW-3644746 View in Salesforce ↗
Fully Resolved
What Happened

VideoAmp was unable to run analyses in your shared clean room, receiving a masked "Error in secure object" error. The clean room had been operating normally for months prior to June 9.

Root Cause

VideoAmp's AWS IAM policy for the role accessing Netflix's S3-backed Iceberg tables included a VPC restriction (aws:SourceVpc condition on the BdpSnowflakeVideoAmp IAM role). When Snowflake's VPC IDs rotated, the new IDs were not on the allowlist, causing all S3 data reads to be blocked — surfaced to the end user as the masked error.

The four VPC IDs added to resolve the issue:

vpc-0d71c4cb01df306c0 vpc-0112c764382c46395 vpc-07ed36a09d3735785 vpc-0598249688350681c
Where Things Stand Today

Fully resolved. The required Snowflake VPC IDs were added to the allowlist and partner access was restored. VideoAmp is running queries successfully.

Next Steps

Netflix to complete VPC allowlist updates across remaining Snowflake accounts using SYSTEM$GET_SNOWFLAKE_PLATFORM_INFO().

Note: Snowflake provides advance notice of VPC changes via release notes and email notifications to account administrators — we recommend ensuring the relevant contacts are subscribed to Snowflake release communications. Snowflake has also submitted a product feature request for a real-time automated notification mechanism; the account team will follow up on its progress.

Case 01380107 — Listing Refresh Timeout Errors

Opened: June 16, 2026 Contact: Alex Bohr (alexbohr@netflix.com) Jira: SNOW-3671107 View in Salesforce ↗
UnblockedFix In Progress
What Happened

Calls to SYSTEM$TRIGGER_LISTING_REFRESH for CLEAN_ROOM_INTERNATIONAL_DATA_SHARE were returning a timeout error targeting Azure West Europe consumers. The error was alarming but misleading — all underlying data refreshes completed successfully with a 100% delivery rate throughout the incident period. No data was lost or delayed to your partners.

Root Cause

A regression in Snowflake's internal cross-region messaging service caused acknowledgement replies from certain deployments to exceed the function's 60-second timeout window. The refresh itself committed and delivered successfully; only the confirmation signal was delayed.

Where Things Stand Today

Your pipelines are operationally unblocked using the catch-and-verify pattern — the pipeline catches the timeout error and confirms success via SYSTEM$LISTING_REFRESH_HISTORY_V2. In parallel, Snowflake Engineering has approved and is actively implementing a permanent product fix: the function will be converted to an async model so it no longer raises a hard error for this condition. Once deployed, the workaround will no longer be needed.

Next Steps
Immediate: No action needed — pipelines are running via the catch-and-verify workaround.

Upcoming: Snowflake will notify you when the product fix is deployed (estimated ~2 weeks from June 26), at which point the workaround can be removed.

Case 01387602 — Inconsistent Row Counts from Shared Clean Room View

Opened: June 24, 2026 Contact: Alex Bohr (alexbohr@netflix.com) No Jira filed View in Salesforce ↗
Resolved
What Happened

A COUNT query returned 176,494 rows while a subsequent INSERT from the same shared view processed only 98,310 rows, appearing to indicate data loss.

Root Cause

No data was lost. The COUNT query was executing against the producer account (US East) and the INSERT against the consumer account (EU Central), which receives data via cross-cloud auto-fulfillment replication. At the time of the queries, the consumer's replicated copy was approximately 2 hours behind the producer — a normal characteristic of asynchronous cross-cloud replication. The INSERT correctly processed every row available in the consumer's view at that moment.

Where Things Stand Today

Snowflake provided a deterministic consistency workflow using AVAILABLE_LISTING_REFRESH_HISTORY to confirm the consumer account has fully received a refresh before executing INSERT operations. Polling for PHASE_NAME = 'COMPLETED' before proceeding eliminates the risk of acting on a partially replicated snapshot.

Next Steps
Netflix to implement the AVAILABLE_LISTING_REFRESH_HISTORY polling check in the pipeline and confirm with Snowflake Support that the workflow meets requirements.

Case 01388550 — SQL Execution Errors on Shared Iceberg Tables (Azure West Europe)

Opened: June 25, 2026 Contact: Raymond Liu (raymondl@netflix.com) Escalated: Severity 1 Jiras: SNOW-3707703, SNOW-1822996 View in Salesforce ↗
Mitigation Applied
What Happened

The TF1 partner consumer account in Azure West Europe began receiving SQL execution errors when querying tables shared via TF1_FTAB_PROD_LISTING. Four tables were affected in a shifting pattern: tf1_episode_agg and tf1_season_agg remained broken, tf1_total_agg newly failed, and tf1_title_agg recovered. The provider side remained fully operational throughout. This case was escalated to Severity 1 with a dedicated support engineer and Senior Escalation Manager assigned.

Root Cause

A cross-cloud data copy process silently wrote zero-byte files to the Azure interop storage layer for the June 24 data partition. When the consumer's query engine attempted to read those files, it received an HTTP range error because the files contained no data. This was a platform-level storage copy failure, not caused by any changes on Netflix's side.

Where Things Stand Today

Snowflake Engineering identified and applied a mitigation. The affected tables have recovered. Snowflake's Storage team is working on a long-term fix to prevent silent failures in the cross-cloud copy pipeline from occurring in the future.

Next Steps
No action needed. If you observe any recurrence of SQL execution errors on the TF1 listing tables, please flag immediately so Engineering can verify the mitigation remains in effect.
